# Database Schema

The Clinical AI Webapp requires a relational database to track patients, source documents, generated outputs, billing summaries, processing runs and user roles.  This document defines the tables, columns, types and relationships using a PostgreSQL dialect.  You should implement these tables via SQLAlchemy models and Alembic migrations.

## Tables

### patients
Stores basic information about each patient and links to their Google Drive folder.

| Column         | Type           | Notes                                               |
|--------------- |--------------- |-----------------------------------------------------|
| id             | UUID (PK)      | Primary key                                         |
| name           | VARCHAR(255)   | Patient’s full name                                 |
| drive_folder_id| VARCHAR(255)   | Google Drive folder ID for this patient             |
| created_at     | TIMESTAMP      | When the patient record was created                 |
| updated_at     | TIMESTAMP      | When the patient record was last updated            |

### source_documents
Represents raw files imported from Google Drive.  Each file is classified by type and associated with a patient.

| Column           | Type           | Notes                                               |
|----------------- |--------------- |-----------------------------------------------------|
| id               | UUID (PK)      | Primary key                                         |
| patient_id       | UUID (FK)      | References `patients.id`                            |
| drive_file_id    | VARCHAR(255)   | Google Drive file ID                                |
| name             | VARCHAR(255)   | File name                                           |
| file_type        | VARCHAR(50)    | Enum: `INTAKE`, `ASSESSMENT`, `ZOOM_NOTE`, `UNKNOWN`|
| uploaded_at      | TIMESTAMP      | When the file was first detected                    |
| processed        | BOOLEAN        | Whether this file has been processed                |
| created_at       | TIMESTAMP      | Record creation timestamp                           |

### output_documents
Represents PDFs generated by the system (summaries, session notes, treatment plans).  Links to the source documents used.

| Column           | Type           | Notes                                               |
|----------------- |--------------- |-----------------------------------------------------|
| id               | UUID (PK)      | Primary key                                         |
| patient_id       | UUID (FK)      | References `patients.id`                            |
| doc_type         | VARCHAR(50)    | Enum: `SUMMARY`, `SESSION_NOTE`, `TREATMENT_PLAN`   |
| drive_file_id    | VARCHAR(255)   | Google Drive file ID of the generated PDF           |
| summary_id       | UUID (FK)      | Nullable.  References `output_documents.id` for the summary used in a treatment plan |
| session_note_id  | UUID (FK)      | Nullable.  References `output_documents.id` for the session note used in a treatment plan |
| created_at       | TIMESTAMP      | When the PDF was generated                          |
| status           | VARCHAR(50)    | Enum: `DRAFT`, `FINAL`, `ERROR`                     |
| error_message    | TEXT           | Nullable.  Error details if generation failed       |

### processed_files
Tracks processed source files to avoid re‑processing the same file.  This is separate from the `processed` flag on `source_documents` to allow fine‑grained tracking across multiple runs.

| Column           | Type           | Notes                                               |
|----------------- |--------------- |-----------------------------------------------------|
| id               | UUID (PK)      | Primary key                                         |
| source_document_id| UUID (FK)     | References `source_documents.id`                    |
| processed_at     | TIMESTAMP      | When the file was processed                         |

### billing_summaries
Stores structured billing information extracted from generated outputs.  Each summary corresponds to one encounter (document) and can be copied into Headway.

| Column               | Type        | Notes                                                 |
|--------------------- |----------- |-------------------------------------------------------|
| id                   | UUID (PK) | Primary key                                           |
| output_document_id   | UUID (FK) | References `output_documents.id`                      |
| patient_name         | VARCHAR(255)| Duplicate of patient name for quick access           |
| date_of_service      | DATE       | Encounter date                                        |
| service_name         | VARCHAR(255)| E.g., `Psychiatric Evaluation`, `Psychotherapy`, etc. |
| icd10_codes          | VARCHAR(255)| Comma‑separated ICD‑10 codes                          |
| cpt_codes            | VARCHAR(255)| Comma‑separated CPT codes with modifier 25 if applied |
| psychotherapy_minutes| INTEGER     | Total psychotherapy minutes                           |
| headway_block        | TEXT        | One‑line billing summary for copy into Headway        |
| reimbursement_notes  | JSONB       | Optional.  Stores reimbursement calculations for evaluation comparisons |
| created_at           | TIMESTAMP   | When the billing summary was generated               |

### processing_runs
Logs each automatic or manual generation attempt.  Contains metadata for debugging and audit purposes.

| Column           | Type           | Notes                                               |
|----------------- |--------------- |-----------------------------------------------------|
| id               | UUID (PK)      | Primary key                                         |
| patient_id       | UUID (FK)      | References `patients.id`                            |
| doc_type         | VARCHAR(50)    | Type of document attempted: `SUMMARY`, etc.         |
| run_at           | TIMESTAMP      | Timestamp of the attempt                             |
| success          | BOOLEAN        | Whether the generation succeeded                     |
| error_message    | TEXT           | Nullable.  Error details if it failed               |

### users
Stores user accounts for authentication and role management.

| Column           | Type        | Notes                                             |
|----------------- |----------- |---------------------------------------------------|
| id               | UUID (PK) | Primary key                                       |
| email            | VARCHAR(255)| User email                                       |
| password_hash    | VARCHAR(255)| Secure password hash (e.g., bcrypt)              |
| role             | VARCHAR(50) | Enum: `ADMIN`, `PROVIDER`                        |
| created_at       | TIMESTAMP   | When the user was created                        |
| updated_at       | TIMESTAMP   | When the user record was last updated            |

### review_statuses
Tracks the review state of generated documents.  This allows drafts to be edited and approved before finalising.

| Column           | Type           | Notes                                               |
|----------------- |--------------- |-----------------------------------------------------|
| id               | UUID (PK)      | Primary key                                         |
| output_document_id| UUID (FK)      | References `output_documents.id`                    |
| reviewer_id      | UUID (FK)      | References `users.id`                               |
| status           | VARCHAR(50)    | Enum: `PENDING`, `APPROVED`, `REJECTED`              |
| comments         | TEXT           | Optional reviewer comments                           |
| updated_at       | TIMESTAMP      | When the review status was last updated             |

## Relationships and indexes

* Each `patient` can have many `source_documents`, `output_documents`, `billing_summaries`, and `processing_runs`.
* Each `output_document` may link to one or more `source_documents` via `summary_id` and `session_note_id`.  Use nullable foreign keys to allow missing references.
* Add indexes on `source_documents.file_type`, `output_documents.doc_type`, `billing_summaries.date_of_service` and `users.email` to optimise common queries.
* Use `ON DELETE CASCADE` for foreign keys from `source_documents` to `patients` and from `output_documents` to `patients` so that deleting a patient removes associated data.

## Notes

* Timestamps should use `timezone=True` for accurate time handling.
* Use `UUID` columns generated by PostgreSQL’s `uuid_generate_v4()` function or by the application when inserting records.
* For multi‑provider support, extend the `patients` table to include clinician assignment and add appropriate foreign keys.

Implementing this schema will provide a solid foundation for tracking patients, files, outputs, billing and review workflows.
